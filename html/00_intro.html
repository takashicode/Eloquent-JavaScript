イントロダクション

私たちは自分の目的に沿ったシステムを作ったつもりでいる。私たちは自分のイメージに沿ってシステムを作っていると思い込んでいる。実際のところ、コンピュータは我々とは似ていない。そこには我々のほんの一面が反映されるにすぎない。それは論理や秩序や規則や明快さに従う一面だ。

Ellen Ullman, Close to the Machine: Technophilia and its Discontents

この本は コンピュータ に指示をすることについて扱う。今時ではコンピュータはねじ回しみたいにありふれている。でもそれよりは多少複雑だし、僕らの言うとおりのことをさせるのが難しいこともある。

もし、君のさせたいタスクが分かりやすくてありふれたものだったら(メールを表示するとか電卓にするとか)、ただアプリを開いて作業にかかればいい。でも、固有で制限がないようなタスクにはちょうどいいアプリケーションがないかもしれない。

ここでプログラミングが登場する。プログラミングとは、プログラムを構築する作業だ。一連の指示を与えてコンピュータに何をすればいいか伝える。コンピュータはまぬけで了見がせまいから、基本的にプログラミングは面倒でイラつくものだ。

幸いなことに、こういう事実を受け入れてまぬけなコンピュータに合わせた厳密な思考法をむしろ楽しんでしまえば、プログラミングは実りある作業になる。手でやったら永遠にかかる作業を数秒で片付けられるし、コンピュータのツールがこれまでできなかったことが可能になる。それから、抽象的な思考についての素晴らしいエクササイズにもなる。

たいていのプログラミングはプログラミング言語で行われる。 プログラミング言語 はコンピュータに指示するための人工的な言語だ。コンピュータとコミュニケートする一番効率的な方法が、人間同士のコミュニケーションによく似てるのは面白い。人間の言語みたいに、コンピュータの言語も単語やフレーズを新しく組み合わせれば、新しい概念を表現することだってできる。

80,90年代のBASICやDOSのような言語ベースのインターフェースが、ある時点まではコンピュータとやりとりする主な方法だった。それはヴィジュアルインターフェースに大きく取って代わられたけど、これは学習はしやすいけどその分自由が制限される。もし君がどこを見ればいいか知ってたら、コンピュータ言語はまだそこ存在しているんだ。そういう言語の1つ、JavaScriptはたいていのモダンWebブラウザーに組み込まれている。だから、どんなデバイスでも使えるといっていい。

この本は、君をそんな言語に親しませて、便利なことや楽しいことができるようになることを目指していく。

プログラミングについて

JavaScriptについて説明する前に、プログラミングの基本原則を紹介したい。プログラミングというのは、ふたをあけてみれば、大変なものだ。基本的なルールはシンプルで明快だ。だけど、そのルールの上に組み上げられていくプログラムはどんどん複雑になっていって、それ自体がルールや複雑さを持つようになっていく。ある意味で、自分で迷路を組み上げて、自分でその中を迷うようなものだ。

この本を読んでいるとものすごくイラつくこともあるだろう。君がプログラミング初心者だったら、消化しないといけない材料がたくさんある。しかもその材料が組み合わされていくから、君はうまくつなぎ合わせることを要求される。

必要な努力をするかどうかは君次第だ。本を追っていくのがむずかしくても、自分の能力について見きってしまうのはやめよう。大丈夫だから。ただやり続ければいい。休憩したり、読み返したり、プログラムの例やエクササイズを理解してるか確認したりすればいい。学ぶというのは大変なことだ。でも学んだことは全部君のものになって、その後の学びをもっと簡単にしてくれる。

活動が無益に至れば、情報を集めよ。情報が無益に至れば、寝てしまえ。

Ursula K. Le Guin, The Left Hand of Darkness

1つのプログラムには色々な面がある。それはプログラマーのタイプしたちょっとしたテキストともいえるし、コンピュータが何をするか方向づける力でもあるし、コンピュータのメモリ上のデータでもありながら、同じメモリへの作業をコントロールもする。プログラムを僕らがよく知っているものに例えても、たいてい十分に表現できない。でも表面的に合うのはマシーンの例えだ。たくさんの独立した部品が関わり合いながら、全体が正しく動くようにしている。僕らは、この部品同士を協力し合わせて全体の動きに貢献させる方法を考えないといけない。

コンピュータとは、そうした形がないマシーンのホストとして動作する物理的なマシーンのことだ。コンピュータ自体はバカみたいに真っ正直なことができるだけだ。それがなぜこんなに便利なのかというと、それをものすごいスピードでやってくれるからだ。プログラムはものすごい数の単純な動作をうまく組み合わせて、とても複雑なことを実現する。

プログラムとは人間の考えでできた建物みたいなものだ。建てるのにコストはかからないし、重さもないし、タイピングする手の中で簡単に大きくなっていく。

でも注意していないとプログラムの大きさと複雑さがコントロール不能になって、作った本人でさえ混乱させられることになる。プログラムを自分のコントロール下に置くことが、プログラミングの大きな課題だ。プログラムがきちんと動いているときは、それは素晴らしい。プログミング技術というのは、複雑さをコントロールするスキルのことだ。すごいプログラムというのは、抑制されている。つまり複雑さの中でもシンプルさが保たれているんだ。

プログラマーの中には、よく理解された少しのテクニックのセットだけ使っていれば、複雑さをうまくコントロールできると信じる人もいる。そういう人たちは厳しいルールを作って("ベストプラクティス")、プログラムがこうあるべきという形式をあらかじめ決めて、その狭くて安全なゾーンに注意深くとどまろうとする。

このやり方は退屈なだけじゃなくて、効果もない。新しい問題には新しい解決方法が要求されることが多い。プログラミングという分野は若いし、成長も速いし、いろいろなアプローチが有効になる余地が十分にある。プログラムデザインでは、ひどいミスをたくさん起こるものだ。やってみて実際にミスってみよう。そうすればミスをより理解できる。良いプログラムがどういう感じかは実践の中で分かってくる。ルールのリストを見ても分かるものじゃないんだ。

なぜ言語が重要か

まず初めに、コンピュータの生まれた頃には、プログラミング言語は存在しなかった。プログラムはこんな見た目をしていた。

00110001 00000000 00000000
00110001 00000001 00000001
00110011 00000001 00000010
01010001 00001011 00000010
00100010 00000010 00001000
01000011 00000001 00000000
01000001 00000001 00000001
00010000 00000010 00000000
01100010 00000000 00000000

これは数字を1から10まで足しあわせて結果をプリントアウトするプログラムだ(1 + 2 + ... + 10 = 55)。これは単純な架空のマシーン上で実行できるかもしれない。初期のコンピュータをプログラムするには、大量のスイッチの並びを正しいポジションにしたり、ボール紙に穴を空けてコンピュータに与えたりしないといけなかった。これがどんなに退屈でエラーが起こりやすいか想像できるだろう。単純なプログラムを書くにも賢さや規律が必要だった。複雑なプログラムを書くなんてほとんど想像もできなかった。

もちろん、手作業でこの難解なビット(1と0)のパターンを打ち込む作業はプログラマーに偉大な魔法使いになったような深淵な気分を与えてくれた。それは仕事の満足という面でも何かしらの価値があったに違いない。

上のプログラムでは1行ごとに1つ指示をしている。人間の言葉ではこのように書ける。

  1. 数字の0をメモリロケーション0に保持する。

  2. 数字の0をメモリロケーション1に保持する。

  3. メモリロケーション1の値をメモリロケーション2に保持する。

  4. 11をメモリロケーション2の値から引く。

  5. メモリロケーション2の値が0だったら9番の指示へ進む。

  6. メモリロケーション1の値をメモリロケーション0へ加える。

  7. メモリロケーション1の値に1を加える。

  8. 指示3へ進む。

  9. メモリロケーション0の値を出力する。

これはビットのごった煮よりは読みやすいが、なんだかあいまいだ。名前を使って、指示やメモリロケーションの数字を置き換えると助けになるだろう。

 Set “total” to 0.
 Set “count” to 1.
[loop]
 Set “compare” to “count”.
 Subtract 11 from “compare”.
 If “compare” is zero, continue at [end].
 Add “count” to “total”.
 Add 1 to “count”.
 Continue at [loop].
[end]
 Output “total”.

これでプログラムがどう動くか分かるだろうか?はじめの2行は2つのメモリロケーションに初期値を与える。totalは計算結果を組み上げるの使われて、countはその時に扱っている数字を追いかける。compareを使っている行が一番変に思えるかもしれない。プログラムはcountが11と等しいか調べて、実行を止められるか決める。ぼくらの空想マシーンは原始的なものだから、数字が0かどうか調べて、それで意思決定することしかできない。だからcompareというメモリロケーションを使ってcount - 11という計算をして、その結果によって意思決定をする。その次の2行は、countがまだ11になってなければ、countの値を結果に足して、countの数字を1つ増やす。

同じプログラムがJavaScriptではこうなる。

let total = 0, count = 1;
while (count <= 10) {
  total += count;
  count += 1;
}
console.log(total);
// → 55

このバージョンにはもう少し前進がみられる。一番大事なのは、プログラムにどうやって行ったり来たりするか指示する必要がなくなっていることだ。そこはwhileの部分がやっておいてくれる。whileは、与えられた条件が満たされている間は、ブロック(波括弧に囲まれた部分)の中身を実行し続ける。その条件とは、count <= 10で、“countが10以下である"という意味だ。もう一時的な値を作って0と比べたりしなくてもいい。そんなのはつまらないディティールでしかなかった。プログラミング言語の威力の1つはこういうつまらないところを代りにやっておいてくれることだ。

プログラムの最後で、while構文が終わった後に、console.logの操作が結果を書き出す。

最後に、もしたまたまrangeとsumという便利な操作があったとしたらプログラムは以下のようになるだろう。ここで、rangeはある範囲(“range”)の数値の集合を作り、sumは数値の集合の和(“sum”)を計算する。

console.log(sum(range(1, 10)));
// → 55

この話の教訓は、同じプログラムも表現によって長くもなれば短くもなるし、読みやすくもなれば読みづらくもなるということだ。はじめのバージョンはとんでもなく見通しが悪かったけど、最後のはほとんど英語だ。log the sum of the range of numbers from 1 to 10(1-10のrangeのsumをlogする)。後の章ではsumやrangeのような操作を定義する方法を学んでいこう。

良いプログラミング言語とは、コンピュータがする動作について、プログラマーが高いレベルで指示できるようにしてくれるものだ。そして、ディティールを省略できるように便利な部品(whileやconsole.lgoのような)を与えてくれて、オリジナルの部品を定義させてくれるし(sumやrange)、その部品を組み合わせるのも簡単にしてくれる。

JavaScriptとは何か?

JavaScriptは1995年に、Netscape Navigator browserのウェブページにプログラムを加えるために導入された。それからずっと主なグラフィックベースのウェブブラウザに採用されている。そして、操作のたびにリロードしないでも良いような、モダンなウェブアプリを可能にしてきた。昔ながらのウェブサイトでも、ページとやり取りできるようにしたり、ページをお利口にしたりしている。

これは重要な点なんだけど、JavaScriptはJavaという名前の言語とはほとんど関係がない。そっくりな名前は、正気の判断ではなくてマーケティング考慮がもたらしたものだ。JavaScriptが導入されたとき、Java言語はゴリ押しされていて人気を得始めてもいた。誰かがこの成功に乗っかりたいと良いと考えた。そうして僕らは名前で行き詰まることになる。

JavaScriptがNetscapeの外でも採用された時、どう動作するべきか決める標準化ドキュメントが書かれた。そうすればJavaScriptをサポートしていると主張している色々なソフトウェアが本当に同じ言語について話していることになる。これは、標準化をしたECMAインターナショナルの名前をとって、ECMAScript標準と呼ばれる。実際のところ、ECMAScriptとJavaScript、どちらを使っても変わらない。同じ言語に付けられた2つの名前だからだ。

JavaScriptについて ひどい ことを言う人もいる。そしてそれはたいてい本当だ。はじめてJavaScriptで何か書く必要に迫られたとき、すぐに嫌になってしまった。書いたことをほとんど何でも受け入れてくれるのに、全然意図したようには解釈してくれないからだ。これは僕が自分で何をしているか分かっていなかったからだ、もちろん。でも、ここには本当の問題もある。つまり、JavaScriptは何を許容するかにかけてはバカみたいに自由なんだ。この設計の元にある思想は初心者にもJavaScriptでのプログラミングを簡単にすることだ。実際のところ、この設計はプログラムの問題を見つけるのを難しくする。というのも、システム自体が見つけて教えてくれないからだ。

とはいえ、この柔軟さにはアドバンテージもある。もっと堅苦しい言語では不可能なワザを使う余地もあるし、後にみていくように(例えばChapter 10で)JavaScriptの欠点を乗り越えるために役立つこともある。JavaScriptを正しく学んで、しばらく取り組んでみてようやく、僕はJavaScriptが本当に 好きになる ということが分かった。

JavaScriptにはいろいろなバージョンがある。ECMAScriptバージョン3はJavaScriptが優位に登りつめたときに広くサポートされていたものだ。それはだいたい2000年から2010年の間のことだった。この間にも、野望に満ちたバージョン4の作業が進められていた。それは過激な改善や拡張をいくつか含むものだった。すでに広く使われている言語に過激な変更を加えるのは政治的にむずかしいことが分って、バージョン4の作業は2008に廃棄された。そして控え目な野望で当たり障りのない改善を加えたバージョン5が2009年に発表された。そして2015年にはバージョン6が登場した。それはバージョン4のために計画されたアイデアも取り込まれたメジャーアップデートだった。以来、毎年スモールアップデートが重ねられている。

言語が進化しているということは、ブラウザもそれについていかないといけないということだ。もし君が古いブラウザを使っているなら全ての機能をサポートしていないかもしれない。JavaScriptの設計者はアップデートした結果、古いプログラムが壊れないように気を付けている。だから新しいブラウザでも古いプログラムは動作する。この本では、2017年版のJavaScriptを使う。

ウェブブラウザはJavaScriptが使われる唯一のプラットフォームではない。MongoDBやCouchDBはJavaScriptをスクリプトとクエリに使う。デスクトップやサーバーのためのプラットフォームでは(特にChapter 20で扱うNode.jsが重要だ)、JavaScriptのプログラミングをブラウザの外でできる。

コードにどう取り組むか

コード とはプログラムを作り上げているテキストのことだ。この本のたいていの章にはたくさんのコードが載っている。僕はコードを読むのも書くのもプログラミング学習では欠かせないと信じている。ただ例をざっと眺めるんじゃなくて、しっかり注意を払って読んで理解してほしい。始めのうちは時間がかかるし混乱するかもしれない。でもすぐにコツがつかめると約束する。エクササイズについても同じことがいえる。実際に機能する回答が書けるまでは理解したつもりになってはいけない。

エクササイズの答えを実際のJavaScript環境で動かしてみることをオススメする。そうすれば自分のやっていることが上手く行くがどうかすぐにフィードバックが得られる。さらに実験したくなってエクササイズのその先まで行ってほしい。

この本をブラウザで読んでいるなら、プログラム例をクリックすると、編集したり実行したりできるようになる。

この本のプログラムを、このウェブサイト以外で実行したいなら、少し注意が必要だ。だいたいの例は独立したもので、どんなJavaScript環境でも動く。でも後の方の章の例は特定の環境(ブラウザーかNode.js)に向けて書かれていて、そこでしか動かない。さらに、多くの章では大きいプログラムを扱っていて、コードの部分部分がお互いに依存していたり、外部のファイルを必要としていたりする。このサイトのsandboxに、スクリプトやデータを含んだZipファイルが置いてある。

この本の概要

この本は、おおむね3つの部分でできている。始めの12章はJavaScript言語そのものについてだ。続く7章はウェブブラウザとそれをJavaScriptで動かす方法についてだ。最後に2の章をNode.jsにさいた。これはJavaScriptを動かすもうひとつの環境だ。

この本では5つの プロジェクトの章 がある。ここでは実際のプログラミングの感じを掴むために大きいプログラムの例を扱っている。出現の順で、配達ロボット、プログラミング言語、プラットフォームゲーム、ピクセルペイントプログラム、動的なウェブサイトだ。

言語自体について扱う12章はJavaScriptの基本的な構造について扱う4章から始まる。そこでは制御構文(先に出たwhileのようなもの)や関数(自分だけの部品)やデータ構造を紹介する。これで、君は基本的なプログラムを書けるようになる。次に5章と6章では、関数とオブジェクトを使ってもっと 抽象的な コードを書いて、複雑さをコントロールするワザを紹介する。

初めてのプロジェクトの章の後、言語自体についての章は、エラー処理とバグ修正、正規表現(テキストを扱うための大事なツールだ)、モジュール性(これも複雑性からの防御)、非同期プログラミング(時間がかかるイベントの処理)と続く。2つ目のプロジェクトの章でこの本の第一部は終わりだ。

第二部の、13章から19章ではブラウザのJavaScriptがアクセスできるツールについて扱う。君はスクリーンに表示したり(14章と17章)、ユーザーの入力に反応したり(15章)、ネットワーク上でやりとりしたり(18章)することを学ぶ。第二部にも2つプロジェクトの章が含まれる。

最後に、20章ではNode.jsについて説明して、21章ではその応用でちょっとしたウェブサイトを組み立てる。

タイポグラフィーの慣習

この本ではmonospacedフォントで書かれたテキストはプログラムの一部を意味する。それだけで完結したものかもしれないし、近くのプログラムの一部を示しているだけかもしれない。プログラムは以下のように書かれる(すでにいくつか見たきたけど)。

function factorial(n) {
  if (n == 0) {
    return 1;
  } else {
    return factorial(n - 1) * n;
  }
}

プログラムの結果を表示するために、予想される出力を続けて書いておくこともある。2つのスラッシュとその前に矢印を付けて表示する。

console.log(factorial(8));
// → 40320
